# 购物中心收益测算软件版本备忘录

日期：2026-05-29

## 1. 这份备忘录的目的

这份文档不是单纯记录 Git 操作，而是记录“购物中心收益测算软件”每个版本做了什么、能实现什么、解决了什么问题、还缺什么。

以后每次更新，都应该在这里补充一条版本记录。这样未来回看时，不只是知道代码改过，还能知道产品能力是怎么一步步长出来的。

## 2. 项目定位

本项目是一个面向城市更新和存量商业资产改造的前期收益判断工具。

它的目标不是替代正式评估报告、造价软件、招商系统或资产管理系统，而是帮助业主、投资人、设计公司或运营方在早期快速回答几个问题：

- 这个老购物中心或存量商业资产是否值得继续改造；
- 改造后可能产生多少租金收入；
- 改造大概需要多少投入；
- 项目每年可能产生多少 NOI；
- 投资回报率和回收期大概处在什么区间；
- 应该推进、谨慎推进，还是暂不建议推进。

## 4. 第一版能实现什么

### 4.1 项目输入

用户可以输入或调整：

- 项目名称；
- 地址或片区；
- 建筑面积；
- 可出租比例；
- 当前客流；
- 当前出租率；
- 城市层级；
- 人口流动；
- 客群结构；
- 商业定位；
- 商圈成熟度；
- 未来片区发展；
- 产业和办公支撑；
- 周边竞品数量；
- 地铁和交通条件；
- 道路展示面；
- 可见性；
- 主力品牌或锚点；
- 业态组合；
- 楼层、单层面积、公区面积；
- 停车位和停车场改造面积；
- 轻改、重改、外立面、机电等改造范围；
- 收购成本、改造成本、其他收入和运营成本比例。

### 4.2 三层测算逻辑

第一版把测算分成三个层级：

- 一级：快速判断，用少量核心参数判断项目大方向。
- 二级：AI 收入测算，加入城市、人口、商圈、交通、竞品、业态和品牌因子。
- 三级：方案成本测算，加入建筑面积、公区、停车、轻改、重改、外立面和机电等工程量因素。

这样做的意义是：用户不需要一开始就填完整资料，可以先粗判，再逐步加深。

### 4.3 收入测算

第一版可以根据以下因素估算有效租金：

- 城市租金基准；
- 人口流入或流出；
- 客群结构；
- 商圈成熟度；
- 未来发展潜力；
- 产业和办公支撑；
- 周边竞品数量；
- 地铁和道路可达性；
- 展示面和可见性；
- 品牌锚点；
- 业态组合；
- 当前客流强度；
- 人工租金调整项。

输出结果包括：

- 市场基准租金；
- 有效租金；
- 年租金收入；
- 可出租面积；
- 租金条件评分；
- 租金信心判断。

### 4.4 成本测算

第一版可以估算以下成本：

- 收购成本；
- 轻改造成本；
- 重改造成本；
- 停车和外立面成本；
- 机电改造成本；
- 其他投入；
- 总投资额。

这个成本模型不是正式造价模型，但可以用于早期判断项目是否值得进入下一轮方案和造价验证。

### 4.5 财务指标

第一版可以输出：

- 年租金收入；
- NOI；
- 总投资；
- ROI；
- 投资回收期；
- 项目评级；
- 推进建议。

ROI 和回收期是当前版本最核心的判断指标。

### 4.6 AI Agent 风格判断

第一版会生成类似顾问判断的文字结果，包括：

- 是否建议推进；
- 支撑推进的理由；
- 主要风险；
- 收入端判断；
- 成本端判断；
- 财务端判断；
- 下一步建议。

这部分的意义是：让软件不仅给数字，还能解释为什么。

### 4.7 敏感性分析

第一版包含基础敏感性分析，用来观察关键假设变化后 ROI 的变化。

适合重点观察：

- 租金变化；
- 出租率变化；
- 改造成本变化；
- 收购成本变化；
- 运营成本变化。

## 5. 第一版的边界和不足

第一版是原型，不是正式评估系统。当前边界如下：

- 没有接入真实地图、商圈、人口和租金数据库；
- 没有接入 AI 模型自动读取项目资料；
- 没有真实竞品数据抓取；
- 没有导出 PDF 或正式报告；
- 没有用户账号和项目保存功能；
- 没有多项目对比；
- 没有现金流年份模型；
- 没有融资结构、税费、退出估值和 IRR；
- 改造成本仍是经验口径，不是工程造价清单；
- 所有参数仍需要用户手动输入或调整。

第一版的价值在于：把产品方向、输入框架、计算口径和输出形态先跑通。

## 7. 版本命名规则

### 7.1 版本号

建议使用：

```text
v主版本.次版本.修订版本
```

规则：

- `v1.0.0`：第一个可演示版本；
- `v1.1.0`：新增一个重要功能；
- `v1.1.1`：修复小问题或微调；
- `v2.0.0`：产品结构、计算模型或使用方式发生明显变化。

### 7.2 适合升级次版本的情况

例如：

- 新增项目保存功能；
- 新增正式报告导出；
- 新增多项目对比；
- 新增地图或商圈数据；
- 新增现金流模型；
- 新增融资结构和 IRR；
- 新增 AI 自动分析项目资料。

### 7.3 适合升级修订版本的情况

例如：

- 修改文案；
- 调整样式；
- 修复输入校验；
- 修正一个公式 bug；
- 优化默认参数；
- 补充说明文档。

## 8. 后续功能路线建议

### 8.1 v1.1 建议方向：报告化

目标：让当前测算结果能变成一份可给别人看的初步判断报告。

建议新增：

- 一键生成报告；
- 打印或导出 PDF；
- 报告封面；
- 项目摘要；
- 核心指标页；
- 风险提示页；
- 下一步建议页。

### 8.2 v1.2 建议方向：模型可信度提升

目标：让测算结果更像一个专业投前模型。

建议新增：

- 更细的租金假设；
- 分业态租金；
- 分楼层租金；
- 稳定期出租率；
- 爬坡期；
- 开办费；
- 租户补贴；
- 物业费和能耗成本；
- 税费；
- 更完整的 NOI 口径。

### 8.3 v1.3 建议方向：多项目对比

目标：让用户能比较不同资产哪个更值得优先推进。

建议新增：

- 多项目列表；
- 项目评分；
- ROI 排序；
- 回收期排序；
- 风险等级对比；
- 投资优先级建议。

### 8.4 v2.0 建议方向：AI 资料分析

目标：从“用户手动填参数”升级为“AI 读取资料并给出初步判断”。

建议新增：

- 上传项目资料；
- 读取租赁台账；
- 读取平面图或面积表；
- 读取改造预算；
- 自动提取关键参数；
- 自动生成初步投资判断。

## 10. 每次更新后的最短操作

每次完成一轮产品更新后，做三件事：

1. 在本备忘录补充版本记录；
2. 提交代码和文档；
3. 推送到 GitHub。

常用命令：

```bash
git status
git add .
git commit -m "说明这次产品更新"
git push
```

一句话原则：

每次更新都要说清楚：这个版本让购物中心收益测算软件多了什么能力、改变了什么判断、留下了什么限制。
